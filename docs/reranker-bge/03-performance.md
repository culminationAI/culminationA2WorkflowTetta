# Производительность и Оптимизация BGE

Реранкер — самая тяжелая часть поискового пайплайна (после самой LLM). Нам нужно выжать из него максимум.

## 1. Магия `use_fp16=True`
На процессорах Apple Silicon (M1-M4) и современных GPU использование половинной точности жизненно важно.
- **Занимаемая память**: С 2.2 GB падает до ~1.1 GB.
- **Скорость**: Рост в 2-3 раза.
- **Точность**: Потери неразличимы глазом.

## 2. Лимиты кандидатов
Реранкер — это узкое горлышко. 
- **Плохо**: Реранжировать 1000 документов (займет 5-10 секунд).
- **Хорошо**: Реранжировать **30-50** документов (займет 100-300 мс).

## 3. Параллелизм (Batch Processing)
Если у тебя несколько GPU или ты на мощном сервере, BGE может обрабатывать пары пачками. Однако на локальном Mac эффективнее обрабатывать весь список за один вызов `reranker.compute_score(all_pairs)`.

## 4. Контроль длинных текстов
Благодаря архитектуре M3, модель поддерживает до 8192 токенов.
- Для запроса: старайся держать его в пределах здравого смысла (до 512 токенов), чтобы не перегружать Cross-Attention.
- Для документа: можно реранжировать целые страницы текста, не опасаясь потери контекста в конце документа.

## 5. Интеграция с Metal (MPS)
Убедись, что PyTorch установлен с поддержкой MPS. 
```python
import torch
if torch.backends.mps.is_available():
    print("Apple Silicon GPU active")
```
BGE автоматически подхватит ускорение, если оно доступно в системе.

## 6. Cold Start (Холодный запуск)
Первый вызов модели всегда долгий (загрузка весов в RAM). 
- В продакшене: держи инстанс `FlagReranker` в памяти как синглтон. 
- Не загружай модель внутри функции поиска.
