# JSON Fundamentals: Язык общения с машинами

Хотя LLM обучались на обычном тексте, JSON стал стандартом де-факто для интеграции ИИ в приложения. В этой главе мы разберем, почему JSON победил XML и YAML, и как ИИ его воспринимает.

## 1. Почему именно JSON?
- **Минимализм**: Меньше лишних символов (тэгов) по сравнению с XML. Это экономит токены и деньги.
- **Строгость**: В отличие от Markdown таблиц, JSON легко парсится стандартными библиотеками любого языка программирования.
- **Иерархия**: Позволяет описывать сложные вложенные объекты (например, "Список дел", где у каждого дела есть "Подзадачи").

## 2. Анатомия JSON для промптинга
ИИ должен знать основные типы:
- **String**: `"текст"` (всегда в двойных кавычках!).
- **Number**: `42` или `3.14`.
- **Boolean**: `true` / `false` (маленькими буквами).
- **Array**: `[1, 2, 3]` — упорядоченный список.
- **Object**: `{"key": "value"}` — набор пар ключ-значение.
- **Null**: `null` — отсутствие значения.

## 3. Как ИИ "видит" JSON
Для модели JSON — это просто последовательность токенов. 
- *Проблема*: Если модель "галлюцинирует" и забывает закрыть скобку `}`, ваш код упадет. 
- *Решение 2026*: Использование **Constrained Decoding** (ограниченного декодирования), когда модель физически не может выдать символ, нарушающий синтаксис JSON.

## 4. Чек-лист хорошего JSON-ответа
1.  **Strict Syntax**: Всегда `"ключи"` в кавычках.
2.  **No Trailing Commas**: Запрещены запятые после последнего элемента в объекте или массиве (частая ошибка моделей!).
3.  **UTF-8**: Правильная обработка неанглийских символов.

## 5. JSON vs Markdown
Никогда не просите модель: "Выдай ответ в Markdown или JSON". Будьте конкретны. 
- Если данные для человека — **Markdown**.
- Если данные для кода — **JSON**.
